# patent-unveil-review v3.0 产品说明

## 1. 产品定位

本 Skill 聚焦工程方法、软件系统、数据处理、控制测算、工程仿真/测算工具类中国发明专利，目标是把已有代码、算法、工程数据、工具原型或方案文档，整理成可供代理人审稿的技术交底书。

它不是完整申请文件起草器，也不替代专利代理师的新颖性、创造性、侵权或 FTO 法律意见。

## 2. 主流程

```text
Step 1  边界与路线分流
        主流程：工程方法/软件系统/数据处理/控制测算
        有限辅助：结构/机械类，仅文字与附图标记支持
        不适用：纯商业规则、管理流程、信息展示等
        ↓
Step 2  项目扫描
        代码、文档、docx/pptx 转 Markdown、工程数据说明
        ↓
Step 3-4 候选专利点与授权可行性初筛
        技术问题、区别特征、技术效果、实施支撑、方案成熟度
        ↓
Step 5  查新与证据留档
        CNIPA EPUB 优先，Google Patents / Espacenet / WIPO / Scholar 复核
        ↓
Step 6  摘要预览
        ↓
Step 7  技术交底书
        默认方法 + 系统/装置 + 电子设备 + 存储介质四客体保护思路
        ↓
Step 8  自检与交付门禁
        PASS / WARN / FAIL；结构有限辅助最高 WARN
        ↓
迭代    合并补充材料或纠正事实，另存时间戳新稿并追加修订记录
```

## 3. v3.0 收缩原则

- 做减法，不做万能专利起草工具。
- 主线只服务工程方法与软件系统类发明专利。
- 结构类不再作为并列主模式；没有正式附图、物理机理和验证依据时，不做创造性判断。
- 不引入需要登录、验证码、API key、OAuth 或商业账号的外部依赖。
- 保留现有稳定工具链：CNIPA EPUB 查新、公开页复核、mermaid 渲染、DOCX 转换、查新 dossier、交付记录与迭代记录。

## 4. 交付边界

主流程交付 `.md` 与 `.docx` 技术交底书，并在案件目录中保留：

- `prior_art_dossier.json` / `prior_art_dossier.md`
- `query_log.md`
- `positive_controls.md`
- `unverified_sources.md`
- `交底书交付检查记录.md`
- 迭代时的 `交底书修订对话记录.md`

PASS 仅表示代理人审稿就绪，不表示可直接提交专利局。WARN 表示可作为阶段稿交付但有明确待补项。FAIL 不得作为定稿交付。
