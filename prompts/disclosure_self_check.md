# 交底书生成后自检（Step 8，内部执行）

自检结果用于**修订正文**，默认**不单独输出自检报告**；用户索要时可单独提供。**不得**将自检清单作为交底书一章写入正文。

## 8.1 逻辑与闭环

- [ ] 技术方案是否形成完整闭环（发现→处理→扩展/输出）？
- [ ] 模块之间关联关系是否清晰？上下游是否衔接？
- [ ] 分支样本（如边界样本、低置信度样本等）是否有明确处理路径？
- [ ] 依赖项（如类别中心、动态阈值等）是否有来源说明？
- [ ] 第五章「技术关键点」与第三章「系统流程」是否呼应？第五章以论点概述为主，技术细节在第三章。

## 8.2 公式与参数一致性

**Step 8 必做（含公式时）**：除符号体例与跨节一致外，须**主动复核公式是否正确、公式逻辑是否与第三章叙述一致**（等同用户会提出的「检查公式和公式逻辑是否有误，有误请调整」）；发现问题**直接改稿**，勿仅提示用户自行修改。

### 符号与体例

- [ ] **符号表（3.4.1）**：全文含公式时是否已设 **3.4.1** 并 **先定义符号**（含义、下标、量纲）；式 (1) 及后文每个符号是否均已在表中定义？
- [ ] **formula manifest**：关键公式是否已进入 `formula_manifest`（字段含 id、number、title、latex、display、dimension_check），正文公式是否与 manifest 的 `latex`/`number` 一致，未临场重拼？
- [ ] **量纲校核**：manifest 中每条关键公式是否含 `dimension_check`；是否避免把有单位的能量状态（如 \(B_{s,t}\)，kWh）误称为无量纲 SOC？
- [ ] **维度下标**：是否存在 `^{cpu}`、`^{mem}`、`^{io}` 等 **上标表示维度** 的写法？若有须改为 `b_{i,\mathrm{cpu}}`、`a_{j,\mathrm{mem}}` 等 **下标 + `\mathrm{}`** 形式（见 **`disclosure_builder.md` §7.7**）
- [ ] **字母多义**：同一字母是否兼指任务侧与节点侧等不同对象？若有须拆分符号（如任务用 \(b\)、节点用 \(g\)）
- [ ] **公式表述一致**：同一物理量在不同公式、段落中是否 **同形同义**（如权重公式、调整系数、匹配分 \(M_{ij}\) 等）
- [ ] **LaTeX 体例**：行内/块级分隔符是否全文统一（`\(...\)`/`\[...\]` 或 `$`/`$$` 二选一）；块级公式是否尽量单行；是否避免 `\text{且}` 等易渲染失败写法
- [ ] **3.5 与符号表**：3.5 参数表「符号」列是否与 3.4.1 **逐字同形**
- [ ] **阈值范围**：与正文一致（如 0.5–1.5、0.8–1.2 等）
- [ ] **参数命名**：全文统一，避免同义不同名混用
- [ ] **实施例数值**：与 3.5 关键技术参数对应，不冲突

### 公式正确性与逻辑（必核）

- [ ] **式面正确性**：各式是否无明显笔误（运算符、括号配对、上下标错位、同一式内符号与符号表不符、缺项/多写项、指数或分母写错等）？
- [ ] **约束与不等式方向**：公式中的 `\leq`/`\geq`/正负号是否与文字含义一致（如「饱和度」「有余量」「限频间隔」等）；多条件「且/或」是否与 3.4 流程分支一致？
- [ ] **公式与流程互推**：3.4.1 主公式（含式 (1)）及衍生式，能否与 **3.4 流程说明**（各步骤 S1—Sn）、**3.3 模块职责** 逐步对应？叙述中的打分、重排、限频、窗口聚合等是否在公式中有体现，**无**「文字一套、公式另一套」？
- [ ] **边界与特殊情形**：空队列、单节点、分母为零、指标缺失、达到上下界等情形，公式或正文是否给出合理处理或说明（至少不与主公式矛盾）？
- [ ] **量纲与取值**：符号表声明的量纲/取值范围是否与公式用法一致（如权重为正、比例在 0—1、计数为非负整数等）？
- [ ] **修订联动**：若修正公式或逻辑，是否已同步改 **3.4 文字**、**3.5 参数**、**第六章实施例** 及 **3.4.1 符号表**（见 §7.7 跨节一致）？

## 8.4 结构类有限辅助专项检查（仅 B 路线）

> 本节仅在 `intake.md` Q1.5 B 路线生效时执行。B 路线自检结果最高为 WARN，不输出 PASS。本节只检查文字、附图标记、结构描述和格式支撑；创造性评估需代理人或发明人自行完成。

Step 8 须完成以下检查项，发现文字和一致性问题应直接改稿；缺少正式附图、物理机理或验证依据时，写入待补项，不得用文字推断替代。

## 8.5 代理人审稿门禁与交付检查记录（必做）

本节用于判断本轮交付是否达到“可供代理人审稿”的基本质量，并把检查结果留在案件目录。**不得**把本节表格、PASS/WARN/FAIL 或自检清单写入交底书正文。

### PASS / WARN / FAIL 定义

| 状态 | 含义 | 是否可作为本轮交付 |
|------|------|--------------------|
| PASS | 已达到代理人审稿就绪：结构完整、区别特征和技术效果闭环、查新结论已写入、格式与图示基本可用 | 可交付给用户用于代理人审稿；不表示可直接提交专利局 |
| WARN | 可作为阶段稿交付，但仍有明确待补项，如实验数据、正式附图、联系人信息、代理人口径确认等 | 可交付，但回复中须列明待补项和风险，不得称为最终完成稿 |
| FAIL | 存在关键缺陷，如无查新结论、核心区别特征不清、技术效果无支撑、正文残留自检/内部流程、缺少交付文件 | 不得作为定稿交付；先修正关键缺陷 |

- [ ] 交底书章节完整：背景、现有技术缺点、技术问题、详细方案、优点、保护点、实施例/其它。
- [ ] 1.1 含检索说明、相关文献/专利和技术对比结论；来源状态、经核验稳定 URL 和未核验清单已在证据包/交接说明中留档；有 A/B/C/D 查新结论并与正文风险表述一致。
- [ ] 案件目录已保存或追加 `prior_art_dossier.json` / `prior_art_dossier.md`、`query_log.md`、`positive_controls.md`、`unverified_sources.md`；若用户仅要求轻量口头探索，已在回复中说明未生成完整证据包。

- [ ] 无编造发明人、实验数据、技术效果、参数来源或现有技术内容；待确认内容已清楚标注给用户。

### 查新证据硬门禁

- **FAIL**：交底书 1.1 中把未核验、不可打开或仅为 CNIPA EPUB 提示路径的 URL 写成已核验公开源；或核心查新文献无来源状态；或没有执行查新却写出 A/B/C 式结论。
- **WARN**：存在关键新近 CNIPA 候选尚未被第三方稳定来源收录，但已列入 `unverified_sources.md` 并把结论降级为 D/partial-D；或阳性对照不足 3 条但已说明检索局限和补检路径。
- **PASS**：高相关文献、阳性对照、最小对比表和 A/B/C/D 结论均有可追溯来源状态；未核验条目不影响核心结论，或已被明确排除/待复核。

### 路线降级规则

- **B 路线（结构有限辅助）**：最高 WARN，附注「本技能对结构类方案不做创造性评估，请代理人重点审查创造性支撑、附图完整性和物理机理依据」。
- **A 路线中方案成熟度 ≤1**：最高 WARN，附注「方案成熟度不足，建议补充原型、工程数据或效果指标后再正式提交」。
- **纯商业/管理/信息展示路线**：不进入交底书定稿门禁，应在 Step 1 或 Step 3-4 停止并说明不适用。

### 留档要求

每次向用户落盘交付 `.md` / `.docx` 后，须在交付文件所在案件目录追加：

- 固定文件名：**`交底书交付检查记录.md`**
- 推荐脚本：

```text
python3 ${CLAUDE_SKILL_DIR}/tools/delivery_check_log.py --case-dir "{案件目录}" --status PASS --summary "{检查摘要}" --pending "{待补项}" --artifacts "{案件名_时间戳.md},{案件名_时间戳.docx}" --checks "{关键检查结果}"
```

`--status` 可为 `PASS`、`WARN` 或 `FAIL`。无法执行脚本时，须手工在该文件末尾追加同结构记录。该记录跟随项目/案件目录走，不写入技能本体，也不写入交底书正文。

## 处理原则

发现问题则**直接修订正文**后再交付。  
**公式类**（主流程）：Step 8 须完成 §8.2「公式正确性与逻辑」核对；有误则改公式并联动修订相关章节，**不要**只向用户报告问题而不改稿（除非缺少必要技术事实、无法从正文与材料推断正确写法）。  
**结构有限辅助**（B 路线）：Step 8 须完成 §8.4 全部检查项；技术闭环缺口、主实施例未收敛、语言清洁和标记一致性应尽量修正，但最终最高 WARN，并列明正式附图、物理机理、创造性评估的待补责任。
**门禁类**：若达到 FAIL，先修正到 WARN 或 PASS 再交付；若只能 WARN，必须在回复和 `交底书交付检查记录.md` 中列明待补项。PASS 仅表示代理人审稿就绪，不表示已具备正式提交专利局的法律结论。
