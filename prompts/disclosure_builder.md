# 技术交底书生成模板（Step 7）

详细章节范例与 **mermaid** 图示模版见同目录 **`template_reference.md`**。

## 7.1 章节结构

### 主流程章节结构（工程方法 / 软件系统 / 数据处理 / 控制测算）

```
注意事项
一、技术背景与现有技术
   （一）现有技术（按技术方向分类，含专利检索结果）
   （二）现有技术存在的缺点
二、本发明所要解决的技术问题
三、本发明技术方案的详细阐述
   （一）方案背景
   （二）系统框图（**mermaid**，如 `flowchart` + `subgraph` 分层；交付前经工具转 PNG；**不要** ASCII 文字框图）
   （三）模块功能说明（聚焦作用与关联关系，非输入输出）
   （四）系统流程说明（**mermaid 流程图**，交付前经工具转 PNG；**不要** ASCII 文字/箭头流程图）
   （五）符号与公式（出现公式或形式化变量时**须设**；**须**先符号与变量定义、再写公式；体例见 **§7.7** 与 **`template_reference.md`**）
   （六）关键技术参数（参数表「符号」列与符号表 **同形**）
四、与现有技术相比，本发明具有哪些优点？
五、本发明的技术关键点和欲保护点是什么？
六、其它（实施例、技术效果、参数示例）
```

**章节编号硬性门禁**：主流程交底书正文使用「一、二、三」和「（一）（二）（三）」组织章节；流程步骤使用 `S1、S2、...`；附图使用「图1、图2、...」。不得生成 `## 1. 技术领域`、`### 2.1 已公开技术概况`、`### 3.4.1 ...` 这类 Markdown 标题编号。编号型列表只用于局部条目，不得作为跨章节标题体系；转换为 Word 时不得进入 Word 自动编号系统，须以字面局部编号显示，避免跨章节续号。

### 附录 A：结构类有限辅助章节结构

> 本附录仅在 `intake.md` Q1.5 B 路线生效时使用。本技能对结构类方案不做专利点挖掘和创造性评估，附图须用户自备，物理机理须由用户提供实验、仿真、标准或工程数据依据；自检结果最高为 WARN。

```
1. 注意事项
2. 一、介绍相关技术背景，描述与本发明技术最相近的现有技术，并说明该现有技术存在的缺点
   - 1.1 现有技术（按技术方向分类，含专利检索结果）
   - 1.2 现有技术存在的缺点
3. 二、针对上述缺点，说明本发明所要解决的技术问题
4. 三、本发明技术方案的详细阐述
   - 3.1 技术方案概述（发明核心思路，1-2段概括）
   - 3.2 结构附图及部件编号说明（附图清单 + 统一编号表，不用 mermaid）
   - 3.3 各部件结构与功能说明（逐部件说明形状、材料、连接关系、位置关系，聚焦结构特征而非功能性描述）
   - 3.4 工作状态/动作过程说明（按状态或工位逐段描述：正常→失效→保护→终态，结合附图标记）
   - 3.5 关键结构参数与设计约束（尺寸关系、公差、材料性能约束等，有公式时遵循 §7.7）
5. 四、与现有技术相比，本发明具有哪些优点？
6. 五、本发明的技术关键点和欲保护点是什么？
7. 六、其它（具体实施方式、技术效果、参数示例）
```

**结构有限辅助 §3.2 撰写要求**：

1. **附图清单**：列出全部附图编号与标题（如「图1 正常插合状态剖面图」「图2 微分离等势旁路状态剖面图」），说明各图视角（剖视、俯视、局部放大等）
2. **统一编号表**：全部件用阿拉伯数字编号，格式为「编号-名称」（如「1-公头壳体、2-母头壳体、3-插针导体」），编号须全文统一、附图与文字互斥（同一标记不在同一附图上同时标文字说明）
3. **附图本身**不由交底书 Markdown 生成，由用户提供 CAD/Visio 线稿或手绘草图；交底书中以文字描述附图内容

**结构有限辅助 §3.3 撰写要求**：

- 逐部件说明时，使用**结构特征**（形状、构造、连接关系、位置关系、材料）而非功能性限定
- 正确示例：「同极等势旁路接触件 6 为弧形弹性导电片，一端固定于插针导体 3 的根部，自由端沿轴向延伸并超出主接触区 5 的末端」
- 错误示例：「旁路件用于在主接触失效时提供等电位保护」（这是功能描述，不是结构特征）
- 每个部件说明中应明确其与相邻部件的**位置关系**和**连接方式**

**结构有限辅助 §3.4 撰写要求**：

- 按**状态/工位**分段描述，每段对应一张或多张附图
- 描述顺序应体现部件之间的**动作顺序**和**位移关系**
- 每段须引用具体附图编号和部件编号（如「参见图2，当主接触对（插针导体 3 与插筒导体 4 构成的主弹性接触区 5）发生微分离时……」）
- **技术闭环必须完整**：从正常状态到最终状态（保护/失效/维护），每个阶段都要有对应描述和附图支撑，不得在中间状态断裂
- **主实施例须收敛**：每个部件在主方案中只选定一种具体形态（如旁路件选定「弧形弹性导电片」），其余变型放入第六章实施例或从属权利要求

**结构有限辅助 §3.5 撰写要求**：

- 结构参数表：列出关键尺寸、间距、行程、公差等设计约束
- 参数须有明确的物理含义和来源依据（实验、仿真、标准规范）
- 无可靠来源的具体数值不要写入，改用设计约束关系（如「旁路接触行程 > 主接触行程 + 安全裕度」）
- 有公式时仍遵循 §7.7 符号与公式体例

**现有技术部分撰写硬性要求**：在「按技术方向分类」的**每一条**现有技术（专利或文献）末尾或表格中，必须给出 **经核验的公开源 URL**（规范与示例见 `prior_art_search.md` 与 `template_reference.md`）。**国知局检索**（`EPUB_HITS_JSON`）中对该专利给出的 **`abstract` 非空时**：文中「技术方案 / 应用 / 局限」类表述**须以摘要理解为前提**（消化后重写，非整段粘贴），**禁止**与摘要明显矛盾或脱离摘要杜撰；详见 **`prior_art_search.md`**「`abstract` 必用」。

**授权论证闭环（全文硬性要求）**：交底书必须能读出以下链条：

```text
现有技术缺点 -> 要解决的技术问题 -> 区别技术特征 -> 作用机理 -> 技术效果 -> 欲保护点
```

- 第一章写“现有技术缺点”时，优先围绕 Step 5 查新中最接近现有技术的不足，不要泛泛写行业痛点。
- 第二章的技术问题须从第一章缺点自然推出，避免写成商业目标或产品愿景。
- 第三章须把可用区别特征写成可实施的步骤、模块、结构、连接关系、参数关系或处理逻辑。
- 第四章的技术效果须由第三章技术特征产生；没有来源或机理支撑的效果，写为“预期可改善/待测试验证”，不得写成已验证事实。
- 第五章保护点须回扣第三章的区别技术特征，不把无实施支撑的愿望、场景或结果写成核心保护点。

**软件/AI 案件特别要求**：算法、模型、商业规则或数据处理特征不能孤立写成抽象规则；须说明其与具体技术场景、硬件/系统模块、输入输出、训练或推理流程的配合关系，以及由此解决的技术问题和产生的技术效果。涉及 AI 模型时，按需要披露模型模块、层/连接关系、训练步骤、关键参数、输入输出和应用领域组合，使本领域技术人员能够实施。

**默认保护组合**：主流程优先按「方法 + 系统/装置 + 电子设备 + 计算机可读存储介质」四客体组织第五章保护点。若查新结果或材料支撑不足，可收缩为「方法 + 系统」或「系统」；若方法/控制创新与结构改进并存，优先把方法/控制作为主线，结构侧写入从属特征、实施例或另案建议。

**诚信与支撑边界**：不得编造实验数据、性能提升比例、技术效果、参数来源或用户未提供的事实。确需表达但资料不足时，标注为“待补充实验/仿真/样机验证”或改为定性技术效果。

**禁止输出**：交底书正文中**不得**包含「自检清单」章节，例如检查项表格或带状态符号的清单列。自检见 `disclosure_self_check.md`，**内部执行**，不写入交付正文。

**禁止仓库/技能脚注**：交付用 Markdown **末尾不得**出现任何指向本技能或示例仓库的说明，例如「本文件为 `patent-unveil-review` / `patent-disclosure-skill` 仓库内教学示例」「不构成任何法律或技术承诺」「虚构教学」「详见 `examples/`」等。正文**止于第六章及此前章节**；**不要**在文末追加斜体免责或仓库署名。

## 7.3 输出文件命名（用户产出目录，必遵循）

在用户目录（如 `outputs/{案件标识}/`）写入交底书 **Markdown / Word** 时，**主文件名须体现正文中的「案件名称」**，且**凡向用户落盘交付的定稿均须带时间戳后缀**（见下第 5 点），避免无区分度的固定名（如一律 `disclosure_draft.md`）。**仓库内 `examples/` 为教学统一可仍用固定名**；用户产出不适用该惯例。

1. **提取**：从 `**案件名称**：` 行取完整发明名称；去掉占位（如 `[待填写]`、`【待填写】`）、首尾空格。
2. **规范化**：删除或替换 Windows 非法字符 `\ / : * ? " < > |` 与换行；连续空格可压为单个空格或删去（中文标题通常无空格亦可）。
3. **长度**：文件名（不含扩展名）建议 **≤ 80 个字符**；超长则**截断**到约 80 字并保留语义完整（例如截在「方法及系统」等结尾词之前），勿用无意义随机串。
4. **定稿命令**：`mermaid_render.py` 的 `-o` / `--docx` 须使用**同一主文件名**，且该主名**必须符合第 5 点**（**案件名 + `_` + 14 位时间戳**）。示例（案件名已按上文规范化、过长已截断时）：

   `python3 tools/mermaid_render.py -i "…草稿.md" -o "一种异构计算环境下基于资源画像与限频重排队的批任务调度方法及系统_20260408143025.md"`

   默认同目录生成同名 `.docx`。过程性中间稿（仅自用、非交付）文件名可自定，但**最终交付**不得省略时间戳。

5. **时间戳后缀（凡落盘交付必遵循）**：凡写入用户产出目录、作为**向用户交付**的技术交底书 **`.md` / `.docx`**，主文件名须为：  
   **`{§7.3 规范化案件名}_{YYYYMMDDHHmmss}.md`** 与同名 **`.docx`**。  
   - **首次定稿**与**迭代合并/纠正**适用**同一规则**；每次交付取**当次落盘时**的**本地时间** 14 位数字（年月日时分秒各 2 位，如 `20260408143025`）。  
   - **不要覆盖**已有交付文件；新一次交付即**新时间戳**、新文件名。用户**明确要求**覆盖某路径时从其意。  
   - **`mermaid_render.py`** 的 `-o` 与可选 `--docx` 须与上述主名一致。  
   - **例外**：（1）仓库内 **`examples/`** 示例可继续固定文件名；（2）用户书面指定文件名时从其意（仍建议保留时间戳以免混淆）。

Agent **落盘时**即采用上述命名，并在回复中写明路径，便于用户对照标题与磁盘文件。

## 7.4 图示要点

### 主流程系统框图与流程图

- **系统框图与流程图**均仅用 **fenced mermaid**（本地 `mmdc` 渲染，不依赖外网 PlantUML）；定稿 Markdown 中必须由 `mermaid_render.py` 追加**可见** `![图示 n](...)` 图片引用，Word 中以 PNG 为准，**无需**再附 ASCII 文字框图
- mermaid 内节点标签用简短中文/数字，避免 ①②③。可另写简短「流程说明」段落概括各步，**不得**用 ASCII 框线箭头代替图示

### 结构有限辅助附图

- **正式附图**须为黑白线稿（CAD/Visio/手绘扫描），由用户或制图人员提供，交底书中以文字描述附图内容
- 交底书 Markdown 中**不强制**使用 mermaid，但可**选用** mermaid 画以下辅助图：
  - 状态转移图（`stateDiagram-v2`）：展示连接器从正常→失效→保护各状态的转移关系
  - 动作时序/位移顺序图：展示各部件的动作先后关系
- 辅助 mermaid 图**不替代**正式结构附图，仅用于帮助代理人理解工作原理
- **CNIPA 附图硬性要求**（详见 `references/structural_patent_requirements.md`）：
  - 黑色墨水线条，不得着色、不得使用照片、不得有灰度阴影或渐变填充
  - 部件编号用阿拉伯数字，用引出线指向对应部件
  - 同一部件在不同附图中编号必须一致
  - 附图标记与说明书文字互斥（同一标记不在附图上同时标文字说明）

## 附录 A.1 结构有限辅助语言规范与权利要求指导

> 仅在 `intake.md` Q1.5 B 路线生效时使用。以下内容只用于文字清洁、附图标记一致性和结构权利要求格式参考，不构成创造性评价。

### 语言清洁要求

结构类交底书容易出现以下**必须清除的**语言问题：

| 问题类型 | 示例 | 改法 |
|---------|------|------|
| 报告式/分析式语言 | 「本交底书围绕……进行检索」「经综合对比分析」 | 删除，专利交底书不是分析报告 |
| AI生成残留 | 「实际申请时可由制图人员……」「不以某一在先申请……作为必要条件」 | 删除，交底书是技术公开文件，不是工作备忘 |
| 不确定/对冲表述 | 「可以是A，也可以是B，还可以是C、D、E」（在主方案中并列过多替代件） | 主方案固定一种，替代形态移入实施例或从属权利要求 |
| 功能性限定滥用 | 「用于实现等电位保护的旁路件」 | 改为结构特征：「弧形弹性导电片，一端固定于插针导体根部，自由端沿轴向延伸」 |
| 参数来源不明 | 「间距为0.216 mm」（无数据来源） | 删除具体数值或补充来源依据，改用设计约束关系 |

### 权利要求撰写指导

**结构类独立权利要求**应优先采用"两段式"写法（前序部分 + 特征部分）：

```
一种[产品名称]，包括[已知结构部件]，其特征在于：
还包括/所述[部件名称]为/设有……[用结构特征限定创新点]
```

**结构特征限定顺序**：形状 → 构造 → 连接关系 → 位置关系 → 材料 → 尺寸比例关系

**避免在独权中**：
- 使用纯功能性限定（如「用于……的部件」）
- 包含方法步骤（结构独权中不写操作动作）
- 引入非必要技术特征（缩窄保护范围）
- 强制要求可选功能（如「旁路断开」若非核心创新则不应放独权）

**从属权利要求金字塔**：从独权出发，逐层增加具体结构细节、材料选择、参数范围、替代形态、附加功能等。

## 7.7 符号与公式体例（必遵）

撰写 **3.4.1**、**3.5** 及全文含 LaTeX 的段落时须遵循；正/反例与符号表示例见 **`template_reference.md` §3.4.1**。

### 公式 manifest 先行

- 出现 3 条以上关键公式、价格/能量/控制边界公式、或用户指出公式错误风险时，须先建立 `formula_manifest`（可从 **`templates/patent_formula_manifest.yaml`** 复制），每条至少含：`id`、`number`、`title`、`latex`、`display`、`dimension_check`。
- 关键公式的 `latex` 字段是唯一公式源；正文 3.4.1 只引用或复制 manifest 中的公式，不得由 Agent 在正文临场重新拼写。
- 公式编号由 manifest 的 `number` 管理；正文不得靠大量空格手动右对齐编号。Word 中带 `\tag{}` 的块级公式由 `md_to_docx.py` 写为无边框两列表格：左列公式居中，右列编号右对齐。
- 每条关键公式必须有量纲或取值校核；例如储能能量状态 \(B_{s,t}\) 的单位为 kWh，不得写成无量纲 SOC。若需要 SOC，应另定义 \(SOC_{s,t}=B_{s,t}/B_s^{\max}\)。

### 符号表先行

- 撰写 **3.4.1** 或全文含 LaTeX 时，**先写符号与变量定义**（可按（1）任务/对象下标、（2）节点/环境下标、（3）标量与向量分组），每项至少含：**符号、含义、下标含义（如 \(i\)=任务、\(j\)=节点）、量纲或取值范围**。
- 其后公式、**3.5 参数表**、**第六章实施例**中出现的符号须与符号表 **同形、同义**。
- **禁止同一字母多义**：例如任务侧权重用 \(b\)，节点侧饱和度应改用 \(g\)、\(h\) 等其它字母，勿任务/节点共用 \(b\) 表示不同物理量。

### 下标与上标

- **资源维度/类型标签**（cpu、mem、io、peak 等）一律写入 **下标**，维度名用 `\mathrm{cpu}`、`\mathrm{mem}` 等正体，例如 `b_{i,\mathrm{cpu}}`、`a_{j,\mathrm{mem}}`。
- **禁止**用 `^{cpu}`、`^{mem}`、`^{io}` 等 **上标** 表示维度或字段名（易被读作幂次，亦不符合专利正文常见写法）。
- **上标仅用于**：幂次、转置、序号、撇号类标记；**下标用于**对象编号与维度标签。

### LaTeX 分隔与写法（全文统一）

- **行内公式**：全文统一 **`\(...\)`** 或 **`$...$`** 二选一，**不得混用**。
- **块级公式**：全文统一 **`\[...\]`** 或 **`$$...$$`** 二选一，**不得混用**。
- **禁止代码样式公式**：正文、表格和符号列中的符号/公式不得用反引号包裹，尤其不要写成 `` `B_{s,t}^{tot}` `` 这类代码样式；应写为 \(B_{s,t}^{tot}\) 或对应的统一分隔符形式。
- **Word 交付**：`.docx` 中的符号与公式必须是 Word 原生可编辑公式（OMML），不得把公式截图、公式 PNG 或代码样式文本作为正式公式交付。Mermaid 图示可以是 PNG，公式不可以。
- **DOCX QA**：Word 生成后必须运行 **`tools/qa_docx_math.py`**；含 manifest 时使用 `--manifest <manifest.yaml>`。该工具是 Word XML 级交付门禁，不只检查公式，还检查未渲染 mermaid 源码（如 `flowchart TB`）、Consolas/代码样式残留、Word 自动编号结构和嵌入媒体数量不足。若发现 `frac{`、`mathrm{`、`\(`、`\)`、`$` 等残留，或 manifest 编号/分式/上下标结构校验失败，或图示源码/自动编号结构进入 DOCX，本轮不得作为定稿交付。
- 比较符优先写 `\leq`、`\geq`（定稿工具可兼容 `\le`/`\ge`，正文仍推荐 `\leq`/`\geq`）。
- 块级公式尽量 **单行写完**；需编号时优先使用 manifest `number` 并在块级公式末尾写 `\tag{1}`，由工具结构化排版编号；不要使用空格对齐。
- 逻辑连接词（「且」「或」）优先写在公式 **外** 的中文叙述中；若必须写入公式内，用 `\land`/`\lor`，**避免** `\text{且}` 等复杂文本命令（定稿渲染易失败）。
