# 工程方法与软件系统类专利撰写参考

本文用于 `patent-unveil-review` 主流程：工程方法、软件系统、数据处理、控制测算、工程仿真/测算工具类中国发明专利。

## 一、方法权利要求撰写要则

### 1.1 步骤应有技术含义

每个步骤必须尽量包含：输入数据类型、处理逻辑、判断条件、输出结果。避免只写「获取数据」「处理数据」「输出结果」等空洞步骤。

### 1.2 避免纯规则/纯算法表述

权利要求整体必须体现「技术手段 -> 技术问题 -> 技术效果」的链条。涉及算法、模型、评分、排序、推荐、商业规则时，必须绑定具体技术场景、技术数据、处理过程和可验证技术效果。

### 1.3 步骤命名规范

使用 S1、S2、S3 或「步骤一、步骤二、步骤三」。每个步骤首句概括技术动作，后续展开具体处理逻辑、阈值、约束、模型或数据结构。

## 二、系统权利要求撰写要则

### 2.1 模块定义

每个模块/单元须说明其功能、输入输出边界，以及与其他模块的数据流或控制流关系。不要只列模块名称清单。

### 2.2 系统与方法对应

系统权利要求中每个模块应能对应方法权利要求中的一个或多个步骤。模块间协同关系应体现方法步骤之间的逻辑顺序，而不是把功能简单并列。

## 三、四客体保护组合

主流程默认采用：

1. 方法权利要求。
2. 系统/装置权利要求。
3. 电子设备权利要求。
4. 计算机可读存储介质权利要求。

四客体组合适用于软件系统、工程测算工具、数据处理流程、控制策略、识别方法等方案。若查新风险较高或材料支撑不足，可收缩为「方法 + 系统」或仅保留最有支撑的客体。

## 四、常见授权风险提示

| 风险点 | 表现 | 应对 |
|--------|------|------|
| 被认定为智力活动规则 | 步骤全是评分、排序、推荐、文书处理 | 绑定具体技术数据、处理过程和技术效果 |
| 步骤过于抽象 | 只写获取数据、分析数据、输出结果 | 展开数据类型、特征、算法、约束和异常处理 |
| 缺乏技术效果 | 只说提高效率、便于管理 | 给出准确率、错误率、耗时、稳定性、约束满足率等技术指标 |
| 方法步骤不完整 | 输入到输出之间缺少中间处理 | 补齐清洗、归一化、特征提取、判断、更新、输出等步骤 |
| 与论文或公开材料冲突 | 申请日前已公开核心方案 | 核对公开时间，收缩到未公开或未充分公开的区别特征 |
| 工程数据不足 | 只有想法或概念描述 | 补充原型、样例数据、仿真或工程验证记录 |

## 五、推荐交底书主线

交底书正文应始终围绕以下链条组织：

```text
现有技术缺点 -> 技术问题 -> 区别技术特征 -> 处理流程/模块协同 -> 技术效果 -> 欲保护点
```

若某个效果无法从流程、模块或参数中推出，应改为「待验证」或删除，不得为了显得可授权而编造效果。
