# 专利工作流沉淀：AI 友好执行规范

本文档沉淀一次完整“候选方向 → 查新 → GPT/外部模型策划 → 工程基础 → 收口审核 → 技术交底书”的专利工作流经验，供所有后续专利工作复用。它不是某一案件的事实记录，而是面向 agent、GPT、Codex、Claude 或其它 AI 协作者的通用执行参考。

## 何时读取

在以下场景中，主 agent 应优先读取本文档：

1. 用户从专利候选池中选择一个新方向，要求“按上次流程走一遍”。
2. 任务包含壹专利、高数图、CNIPA、Google Patents、GPT/Claude 咨询和交底书交付的组合流程。
3. 高相关现有技术风险很高，需要决定是否放弃宽方向、收缩窄方向或暂停。
4. 用户希望把某轮经验沉淀进 `patent-unveil-review` 技能。
5. agent 不确定当前应继续查新、问 GPT、补工程材料，还是进入交底书撰写。

## 总原则

专利工作不是“写出一个看起来像专利的文档”，而是建立可追溯的决策链：

```text
候选方向
→ 初步技术问题
→ 查新证据
→ 高风险文献对比
→ 可用区别特征
→ 工程/实验支撑
→ GPT/外部模型策划与收口
→ 技术交底书
→ WARN/PASS/FAIL 交付记录
```

AI 协作者的主要价值是加速检索、整理、对比、方案收紧、工程验证和文档生成；不能把 GPT、Claude、商业库摘要或单轮检索结果当作专利代理人结论。

正式交底书正文必须与工作流记录分离。公式转换说明、DOCX QA 说明、商业库核验状态、检索式、阳性对照、案件证据包位置、Agent/脚本名等内容只能写入交接说明、交付检查记录或证据包；不得写入交底书正文。若发现“为避免 Word 交底书中出现公式残片……”“详细检索记录、阳性对照和来源状态已另行保存在案件证据包中……”这类自述，属于定稿 FAIL，应直接删出正文并转移到交接材料。

## 工作流的四个闭环

### 1. 证据闭环

每条现有技术都必须有明确来源状态。来源状态比“看起来像专利数据库”更重要。

| 来源状态 | 可用于什么 | 不可用于什么 |
|---|---|---|
| `public_source_verified` / `third_party_verified` | 写入交底书 1.1 的公开引用 | 直接得出授权结论 |
| `official_pss_verified` / `official_detail_opened` | 高置信官方复核 | 替代代理人权利要求分析 |
| `official_cnipa_epub_detail_opened` / `official_egaz_page_images_archived` | 官方详情页或官方页图复核、暂停条件核对 | 冒充 PSS 复核或原始 PDF |
| `official_claims_transcribed_from_page_images` | 在页图基础上形成可读权利要求对比 | 当作 CNIPA 可复制文本或代理人法律意见 |
| `cnipa_result_page_parsed` | 候选发现、待复核清单 | 稳定 URL、正文公开引用 |
| `commercial_db_discovered` | 商业库发现候选 | 正文公开引用 |
| `commercial_db_content_checked` | 商业库摘要/著录项内部对比 | 正文公开引用、PASS 依据 |
| `third_party_not_indexed` | 未被稳定源收录的风险提示 | 低估风险或忽略 |
| `official_pss_blocked_in_current_environment` / `official_pdf_download_captcha_pending` | 记录当前环境或验证码边界，指导人工补尾巴 | 降低风险等级或标记已核验 |

最低留档：

- `prior_art_dossier.json`
- `prior_art_dossier.md`
- `query_log.md`
- `positive_controls.md`
- `unverified_sources.md`
- claim chart 或最小对比表

#### 国内官方站点访问与材料分级

CNIPA PSS、EPUB、EGAZ 等国内官方站点对网络出口、代理、WebVPN、浏览器环境和验证码较敏感。若出现 `HTTP 412`、空白页、脚本资源缺失、验证码异常或下载接口失败，优先判断为当前环境受阻，而不是判断专利不存在。

处理规则：

- 学校 WebVPN 或数据库远程访问适合壹专利、高数图等商业库，不适合作为 PSS 官方复核的默认入口。
- PSS 卡住时，建议用户用本机 Chrome/Edge、关闭全局 VPN/代理，或对 `*.cnipa.gov.cn`、`*.cponline.cnipa.gov.cn` 设置 DIRECT；必要时用国内手机热点直连。
- EPUB 官方详情页、EGAZ 官方页图、EGAZ 原始 PDF、PSS 详情页要分开记录。拿到 EPUB/EGAZ 页图可以闭环权利要求图像核对和暂停条件，但不能升级成 `official_pss_verified`。
- 若 EGAZ 原始 PDF 下载需要验证码，agent 不应绕过；记录为 `official_pdf_download_captcha_pending`，并把官方页图、页图 manifest、人工转写文本和重建 PDF留档。
- 交底书是否需要修订取决于风险结论是否实质变化。若只是“PSS 仍阻断，但 EPUB/EGAZ 官方页图已证明暂停条件未触发”，且交底书已写 WARN 和相同边界，通常只更新案件证据包和 HANDOFF，不必重复生成新稿。

### 2. 方向闭环

宽方向通常是用来暴露风险的，不一定是用来写独权的。

执行时先允许宽泛描述存在，但查新后必须做收缩：

```text
宽方向是否已被高相关文献击穿？
→ 若是，明确放弃宽独权
→ 找仍未被完整覆盖的必要组合
→ 把题名、独权主线、工程实施和技术效果都收缩到该组合
```

AI 文档中应固定写出：

- `弃用方向`
- `保留方向`
- `必须保留的独权特征`
- `不写入独权的禁区`
- `暂停条件`

### 3. 工程支撑闭环

如果方案属于工程方法、软件系统、数据处理或控制测算类，不能只停留在概念层。至少应形成一个最小可复现实施例。

推荐工程产物：

| 类型 | 最小要求 |
|---|---|
| 配置 | 固定随机种子、场景规模、时间步长、阈值、输出路径 |
| 数据 | 输入、真值、中间结果、输出结果均有 CSV/JSON |
| 图件 | 拓扑图、理论模板、实测矩阵、轨迹、匹配分数、消融结果 |
| 报告 | Markdown/JSON 报告，首页写明“最小可复现实施例”边界 |
| 测试 | 检查关键文件存在、字段合理、JSON 可解析、核心结果可复现 |
| 数据字典 | 字段、单位、主键/外键、时间索引、解释边界 |

工程材料的目标不是证明“必然授权”，而是支撑：

```text
区别特征可实施
→ 作用机理可解释
→ 技术效果至少有最小样例支撑
→ 代理人能基于材料继续审稿
```

### 4. 交付闭环

交底书交付不能只看文件是否生成。必须形成交付门禁：

- `.md` 与 `.docx` 同时交付；
- 文件名包含案件名和 14 位时间戳；
- mermaid 图已渲染为 PNG 并进入 Word；
- 公式为 Word 可编辑 OMML，含 manifest 时通过 QA；
- 正文无内部提示词、技能名、脚本名、GPT 工作过程或自检清单；
- 交付状态为 PASS/WARN/FAIL；
- 案件目录追加 `交底书交付检查记录.md`。

如果关键高风险文献未复核，交付最高为 WARN。

## GPT/Claude 的正确用法

GPT/Claude 适合做：

- 技术问题重构；
- 权利要求主线收缩；
- 高风险文献压测；
- 工程基础目标设计；
- 交底书前的收口审核；
- 发现“看似可写但会被现有技术打穿”的表述。

GPT/Claude 不适合直接做：

- 正式新颖性/创造性/FTO 结论；
- 未核验文献的最终解释；
- 替代 CNIPA PSS、Google Patents、Espacenet 或代理人权利要求书复核；
- 直接决定 PASS。

每次咨询应落盘：

- `consultation_request_YYYYMMDDHHmm_*.md`
- `consultation_response_YYYYMMDDHHmm_*.md`

咨询请求应包含：

```text
1. 当前候选方向
2. 已确认的最接近现有技术
3. 未确认的高风险来源
4. 想让 GPT 判断的具体问题
5. 明确提醒：GPT 不是代理人，不能输出正式法律结论
6. 要求输出：保留/放弃方向、收缩题名、独权必保留特征、独权禁区、工程基础目标、暂停条件
```

## 高风险未核验文献的处理

当出现一个高风险但未能稳定复核的文献时，不能简单忽略，也不能把摘要当权利要求。

必须做三件事：

1. 记录为 `unverified_sources.md`；
2. 在 README/HANDOFF/交底书注意事项中写 WARN；
3. 写明暂停条件。

暂停条件应具体到特征组合，而不是写成“如果该文献很相关则暂停”。

模板：

```text
若 [文献号] 权利要求已经覆盖：
[必要特征 1]
+ [必要特征 2]
+ [必要特征 3]
则本方向不作为主案推进，仅保留为从属模块、实施例或放弃。
```

## AI 友好文档规范

为了让后续 agent 能稳定接手，所有案件目录中的关键文档应具备以下结构。

### README.md

必须包含：

- 当前题名；
- 当前状态 token；
- 核心假设；
- 证据等级；
- 已核验高风险文献；
- 最大未决风险；
- 目录结构；
- 下一步。

推荐状态 token：

```text
INTAKE
SCAN_DONE
PRIOR_ART_PARTIAL_D
PRIOR_ART_WARN
WIDE_DIRECTION_ABANDONED
NARROW_DIRECTION_SELECTED
ENGINEERING_FOUNDATION_REQUESTED
ENGINEERING_FOUNDATION_DONE
ENGINEERING_FOUNDATION_REVISED
GPT_CLOSING_WARN_OK
DISCLOSURE_DRAFT_DELIVERED_WARN
DISCLOSURE_PASS_READY_FOR_ATTORNEY_REVIEW
BLOCKED_BY_HIGH_RISK_PRIOR_ART
```

### HANDOFF.md

必须包含：

- 一句话目标；
- 用户偏好的分工；
- 当前结论；
- 证据等级；
- 最大未决风险；
- 必须保留的技术核心；
- 不要保护的泛化说法；
- 已落盘资料；
- 下一步最小任务；
- 暂停条件。

### prior_art_dossier.md

必须包含：

- 结论边界；
- 来源状态统计；
- 最小对比表；
- 查新后推荐方向；
- 必须保留的独权特征；
- 暂停条件。

### claim_chart.md

建议按特征列，而不是按文献长段落写：

| 文献 | 太阳角/核心输入 | 拓扑/数据结构 | 理论模板 | 实测矩阵 | 空间轨迹 | 类型区分 | 主要用途 |
|---|---|---|---|---|---|---|---|

这种表格更适合 agent 判断“完整组合是否已被覆盖”。

### engineering summary

必须能回答：

```text
实现了什么？
输出了什么？
怎么复现？
测试结果是什么？
哪些效果可主张？
哪些效果只能弱写？
哪些风险仍未解决？
```

## 常见失败模式

### 失败模式 1：把商业库结果当公开引用

商业库可以发现候选、阅读摘要、反查公开号，但不能直接作为交底书公开引用。写入交底书前必须有稳定公开源或官方复核。

### 失败模式 2：只问 GPT “能不能申请”

这种问题会得到泛泛回答。应改问：

```text
请基于以下高相关文献，判断宽方向是否应放弃；
若继续，只能保留哪些必要组合；
请列独权必保留特征、独权禁区、工程支撑目标和暂停条件。
```

### 失败模式 3：工程实验过于完美

最小实验可以用于说明可实施，但不能把 5 类样本上的 100% 准确率写成强授权效果。应优先写：

- 匹配间隔；
- 轨迹误差；
- 时相一致性误差；
- 定位误差；
- 支撑区域稳定性；
- 消融后混淆模式。

### 失败模式 4：交底书里泄露内部过程

交底书正文不得出现：

- GPT/Claude/Codex 工作过程；
- WebSearch；
- 脚本名；
- 自检清单；
- 技能仓库路径；
- “教学示例”“虚构示例”等元说明；
- 未经公开复核的商业库链接。

### 失败模式 5：只写结果，不写作用机理

专利交底书中必须能读出：

```text
现有技术缺点
→ 要解决的技术问题
→ 区别技术特征
→ 作用机理
→ 技术效果
→ 欲保护点
```

如果技术效果不能回扣到区别特征，应删除或改成“待验证”。

## 新候选池启动模板

当用户从候选池捞取新方向时，可按以下流程执行。

### Agent 执行指令

```text
1. 建立案件目录，先写 README.md、HANDOFF.md、evidence_todo.md。
2. 把候选方向拆成宽题名、核心技术问题、可能区别特征、已知材料、待补材料。
3. 用公开源和商业库分别查新：商业库只作发现/内容核对，公开引用必须复核。
4. 形成 prior_art_dossier、positive_controls、unverified_sources、claim_chart。
5. 若宽方向被打穿，明确放弃宽独权，并收缩到窄组合。
6. 准备 GPT/Claude 咨询请求，让外部模型给出题名、独权必保留特征、独权禁区、工程基础目标和暂停条件。
7. 根据收口后的工程目标做最小可复现实验或最低可实施材料。
8. 再请 GPT/Claude 做工程基础收口审核。
9. 审核通过后，用 disclosure_builder 生成带 WARN/PASS 状态的交底书初稿。
10. 交付 .md + .docx + 检查记录，不覆盖旧稿。
```

### 外部模型咨询问题骨架

```text
你不是专利代理人，请不要给正式新颖性、创造性、FTO 或授权结论。

我正在评估一个中国发明专利候选方向：
[候选题名/技术摘要]

已知材料：
[代码/论文/工程方案/实验/数据]

阶段性查新发现：
1. [高相关文献 A：相同点、不同点、来源状态、链接]
2. [高相关文献 B：相同点、不同点、来源状态、链接]
3. [未核验高风险文献：已知摘要、未核验字段、暂停风险]

请按以下格式审查：
1. 宽方向是否应放弃，原因是什么；
2. 若保留，最窄可防守的发明主线是什么；
3. 建议题名；
4. 独权必须保留的 5-8 个技术特征；
5. 绝不能写入独权的禁区；
6. 与最接近现有技术的区别特征；
7. 技术效果应如何写，哪些只能弱写；
8. 进入交底书前需要补哪些工程材料；
9. 暂停条件；
10. 给 Codex 的可执行工程产物清单。
```

## 更新技能时的建议

若未来继续打磨 `patent-unveil-review`，优先考虑把以下能力固化为 prompt 或脚本：

1. 案件目录模板生成器；
2. 来源状态标准化 schema；
3. claim chart 自动骨架；
4. GPT 咨询请求生成器；
5. 工程基础目标清单生成器；
6. 交底书正文内部流程残留扫描器；
7. WARN/PASS 交付状态解释器；
8. 候选池批量初筛表格。
