# 正式交底书正文清洁规则

适用场景：生成、修订或 QA 技术交底书 `.md` / `.docx` 定稿时，防止内部流程、工具说明、证据管理说明或 AI 自述进入正式正文。

## 核心原则

正式交底书是给发明人、专利代理人和后续申请文件撰写使用的技术公开材料，不是工作流说明、生成日志、查新证据包索引或工具使用说明。正文只写技术背景、相近公开技术、技术问题、技术方案、技术效果、保护点和实施例。

## 正文禁止内容

以下内容不得写入正式交底书正文：

- Agent、Codex、Claude、WebSearch、Playwright、脚本名、skill 仓库名、提示词、自检清单、QA 过程。
- “为避免 Word 交底书中出现公式残片……”“以下公式均以标准 LaTeX 源写入中间稿……”“公式编号作为正文标签管理……”等生成链路说明。
- “详细检索记录、阳性对照和来源状态已另行保存在案件证据包中……”“交底书正文仅摘录……”等交接说明。
- 商业库会话 URL、学校 VPN/session URL、截图编号、检索式、命中数、`query_log`、`prior_art_dossier`、`positive_controls`、`unverified_sources` 等证据包字段。
- `commercial_db_discovered`、`commercial_db_content_checked`、`public_source_verified` 等来源状态标签，以及“壹专利已完成商业库内容核验；正式公开引用仍建议……”这类状态解释。

## 应放置的位置

| 内容 | 放置位置 |
|---|---|
| 检索式、平台、命中数、截图编号 | `query_log.md` / `prior_art_dossier.md` |
| 阳性对照、最小对比表、来源状态 | `positive_controls.md` / `prior_art_dossier.md` |
| 商业库内容核验状态、公开源补齐边界 | `交底书交接说明_*.md` / `交底书交付检查记录.md` |
| Word/公式/图示/DOCX QA 结果 | `交底书交付检查记录.md` |
| 代理人或用户需复核事项 | 交接说明或交付检查记录 |

## 正文推荐写法

1.1.2 写相近公开技术时，使用技术对比口径：

```text
现有公开技术中，与本发明较为接近的方向包括……。下表列举的公开文献均涉及上述一个或多个方向，但其技术目标、处理对象或计算链条与本发明仍存在差异。
```

表格推荐列：

```text
公开文献 | 技术方向 | 与本发明的接近点 | 与本发明的主要区别
```

若必须提醒“尚需公开源复核”或“商业库已完成内容检查”，放入交接说明，不放入正文。

## QA 要求

定稿 DOCX 除常规公式/图示 QA 外，应运行正式正文清洁检查：

```bash
python3 tools/qa_docx_math.py outputs/case/交底书.docx --check-formal-text
```

通过 `mermaid_render.py` 生成正式 Word 时默认执行该检查；仅在诊断非正式中间稿时才可显式关闭。
